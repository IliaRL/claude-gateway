<action>
Append to `AIClient2API/OPERATION.md`:

**Section 3: Troubleshooting**

Format as a symptom → diagnostic → fix table covering the 3 most common failure modes:

**Symptom 1: ECONNREFUSED (proxy not running)**
- Symptom: `curl` to :3000 returns `Connection refused`
- Diagnostic: `ps aux | grep "master.js"` — confirms proxy not running
- Fix: `./scripts/safe-restart.sh` — enforces 2GB RAM floor before starting
- Alternative: `start-proxies` from any shell (same effect)

**Symptom 2: 429 / no healthy provider**
- Symptom: `{"error":"no healthy provider"}` or `pnpm run smoke` shows all FAIL
- Diagnostic: `curl -sf -H "Authorization: Bearer $AICLIENT_TOKEN" http://127.0.0.1:3000/provider_health | python3 -m json.tool`
- Fix: Wait for cooldown to expire (`RATE_LIMIT_COOLDOWN_MS` in config.json, default ~60s) or check which provider hit quota

**Symptom 3: Wrong model / unexpected fallback**
- Symptom: Request routed to wrong provider, or response from unexpected model
- Diagnostic: Check `configs/config.json → modelFallbackMapping` and `providerFallbackChain` for the requested model ID
- Check proxy log: `tail -50 ~/MASTER-C/AIClient2API/logs/app-$(date +%Y-%m-%d).log`
- Fix: Verify the model ID string exactly matches the catalog entry in `configs/model-catalog.json`

Add a "Log locations" subsection:
- Gateway log: `~/MASTER-C/AIClient2API/logs/app-$(date +%Y-%m-%d).log`
- Test output: `pnpm test --verbose` for detailed diagnostics

End with a "Complete restart procedure" block covering the safe-restart.sh command.
</action>
