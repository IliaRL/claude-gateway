# Plan 02-04: DOC-02 — Write docs/SYSTEM-OVERVIEW.md

## Objective

Create `MASTER-C/docs/SYSTEM-OVERVIEW.md` — the architecture reference that explains how
all parts of the 2-tier system connect: traffic flow, provider selection logic,
logging/retry/timeout paths, and security-sensitive areas.

## Context

`SYSTEM-OVERVIEW.md` does not exist in `MASTER-C/docs/`. This doc fills the gap
between the high-level `ARCHITECTURE.md` and the detailed source code. It should be the
first doc a new operator reads after GETTING-STARTED.md.

Key architecture facts from research:
- Traffic: Claude Code → env injection (ANTHROPIC_BASE_URL) → AIClient2API :3000 → external provider
- Provider selection: `selectProviderWithFallback()` in `provider-pool-manager.js`, filters `isHealthy && !isDisabled && !needsRefresh`, score-based (1e18 = never selected), cascades through `providerFallbackChain`
- Retry: `REQUEST_MAX_RETRIES` + `CREDENTIAL_SWITCH_MAX_RETRIES` in `configs/config.json`
- Cooldown: `RATE_LIMIT_COOLDOWN_MS` / `RATE_LIMIT_COOLDOWN_MAX_MS` for 429 handling
- Logs: `~/MASTER-C/AIClient2API/logs/app-YYYY-MM-DD.log` (date-stamped)
- Security: Credentials in `MASTER-C/Credentials/`, live tokens in `provider_pools.json` (never commit), exec calls in `update-api.js` (git ops, system-controlled inputs)

**Section 2: Provider Selection Logic**

Explain in prose + pseudocode:

**Step 1: Model → Provider mapping**
Requested model ID is looked up in `configs/config.json → modelFallbackMapping`:
- If found: uses the mapped canonical model + falls back through `providerFallbackChain`
- If not found: uses model as-is and tries each provider in `providerFallbackChain` order

**Step 2: Pool filtering** (`selectProvider()` in `provider-pool-manager.js`):
Only accounts passing all three conditions are candidates:
- `isHealthy === true` — not in an error state
- `isDisabled !== true` — not administratively disabled
- `needsRefresh !== true` — OAuth token is valid

**Step 3: Score-based selection**
Candidates are scored. Accounts in cooldown or with recent errors score `1e18` (never selected). The lowest-score healthy account wins.

**Step 4: Cooldown handling**
When a provider returns 429:
- That provider enters cooldown for `RATE_LIMIT_COOLDOWN_MS` (+ jitter up to `RATE_LIMIT_COOLDOWN_JITTER_MS`)
- Cooldown escalates up to `RATE_LIMIT_COOLDOWN_MAX_MS` for repeated 429s
- During cooldown, the provider is skipped and the next in the fallback chain is tried

**Step 5: Cascade**
If all accounts for a provider fail or are in cooldown, `selectProviderWithFallback()` moves to the next provider in `providerFallbackChain` for the requested model.

Include a small config excerpt showing providerFallbackChain structure.

**Section 3: Logging, Retry, and Timeout**

**Logs:**
- Location: `~/MASTER-C/AIClient2API/logs/app-YYYY-MM-DD.log` (date-stamped, rotated daily)
- Controlled by: `LOG_LEVEL`, `LOG_DIR`, `LOG_MAX_FILE_SIZE`, `LOG_MAX_FILES` in `configs/config.json`
- Contains: per-request provider selection, response status, latency, errors
- Note: Sensitive token values are NOT logged — provider credentials are redacted

**Retry behavior:**
- `REQUEST_MAX_RETRIES` — number of retry attempts per request before escalating to next provider
- `CREDENTIAL_SWITCH_MAX_RETRIES` — how many times to try switching credentials within a provider pool
- On 429: provider enters cooldown, retry moves to next provider in fallback chain

**Timeout:**
- `STREAM_TIMEOUT_MS` — SSE stream timeout. If the stream stalls for longer than this, the request fails and the fallback chain is tried.

**Startup behavior:**
- `startupRun: false` in `configs/config.json` — prevents startup health-check storms (do not change this)
- Startup order matters for memory: see `scripts/safe-restart.sh` for the correct restart sequence

Read the actual config.json to confirm exact key names and include their default values.
