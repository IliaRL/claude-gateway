Where should we enforce Claude compatibility?

To fix output quality and syntax compatibility, we have a few architectural options for how the gateway can intervene.

A

Strict Output Interceptor (Middleware)

A streaming parser sits between the model and Claude Code. If a model starts outputting wrong syntax (e.g., markdown JSON instead of XML), the interceptor catches it and translates it into Claude XML on the fly.

Model: ```json { "tool": "ls" } ```
Gateway: <invoke name="ls"> ... </invoke>
B

Model-Specific Prompt Injection

We build a "prompt augmentation" layer. Before the request goes to the provider, we inject very strong, model-specific instructions (e.g. "You are Qwen. Do not use JSON. Use this XML format: ..."). Relies on the model following instructions.

C

Continuous E2E Validation Suite

Instead of hacking the stream, we build a rigorous test suite that runs automated Claude Code scenarios against all models. If a model fails the compatibility tests, it gets automatically marked as "unsupported" or downgraded until its prompt is fixed.
